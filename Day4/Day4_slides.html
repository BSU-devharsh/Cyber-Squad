How the web talks

Requests, responses, and statelessness

The web runs on a request-response model: your browser (the client) sends an HTTP request to a server, and the server sends back a response, usually HTML. HTTP is stateless, meaning each request stands alone, so sites use cookies and tokens to remember who you are between requests. Requests use verbs such as GET to fetch and POST to send data, and a URL names the scheme, host, path, and query parameters. HTTPS wraps all of this in TLS encryption. Most web attacks target the moment when the server turns your input into an action, such as a database query.

What is a web application?

Software that lives on a server and runs in your browser

A web application is software that runs in a browser and communicates with a server over the internet. Unlike a desktop app installed on your computer, a web app lives on a remote server; your browser downloads and renders the interface. Examples are Gmail, Instagram, your school's grading portal, and online banking. The browser is the client; the machine serving the content is the server. Most attacks in this unit target the moment when the server takes input from the browser and acts on it, for example running a database query with a user's search term.

TA note: Ask students to name 5 web apps they used today. Every one of them has a server processing their input. That is the attack surface.

How a URL works

The address of a resource on the web

A URL (Uniform Resource Locator) is the address of a resource on the web. It has four parts. The scheme (https) says which protocol to use. The host (example.com) is the server's domain name. The path (/search) is which page or resource to load. The query string (?q=cats&page=2) passes extra data to the server as key=value pairs. Understanding the URL is critical for web security: query parameters are user input that the server acts on, cookies are sent with every request, and the difference between http and https determines whether the data is encrypted in transit.

TA note: Open a real browser and navigate to google.com/search?q=cyber+security. Point to each part of the URL in the address bar.

SQL injection

When input becomes a command

Web apps store data in databases queried with SQL. SQL injection happens when an application builds a query by gluing untrusted user input directly into the command string, so an attacker can insert their own SQL. A classic example is typing a value like quote OR 1=1 into a login field, which makes the condition always true and can bypass authentication or dump every row. The fix is parameterized queries, also called prepared statements, where the input is bound as data to a placeholder and the database never interprets it as code. Validating input adds a second layer, but parameterization is the real cure.

TA note: Type this on the board: SELECT * FROM users WHERE name=''+OR+1=1--'. Walk through why this returns every row: the OR 1=1 is always true, the -- comments out the rest. Then show the parameterized fix: cursor.execute('SELECT * FROM users WHERE name = ?', (name,)). The input is never interpreted as code.

Cross-site scripting

Running attacker code in a browser

Cross-site scripting, or XSS, happens when a site takes user input and places it into a page without cleaning it, so an attacker can inject a script that runs in other visitors browsers. Stored XSS saves the malicious script on the server (for example in a comment) so it hits everyone who views the page, while reflected XSS bounces it back in a crafted link. The injected script runs with the victim trust in the site and can steal session cookies or take actions as the user. Defenses are to encode output so input is shown as text not code, validate input, and apply a Content Security Policy.

Decoding hidden data

Cookies, tokens, and encodings

Web applications constantly move small pieces of data through cookies, headers, and URL parameters, and that data is often encoded rather than encrypted. Session cookies prove you are logged in, and tokens carry claims about you; many are Base64-encoded, which looks scrambled but can be decoded by anyone. Capture The Flag web challenges lean on this by hiding flags inside cookies, comments, or encoded parameters that you recover with the same Base64 and inspection skills used in real testing. Reading these values with browser developer tools is a core beginner skill, and it shows why sensitive data must be encrypted and signed, not merely encoded.

Large language models

AI that follows instructions

A large language model (LLM) is an AI trained on enormous amounts of text that responds by predicting the most likely next words, one token at a time. Because it follows instructions written in plain language, it can summarize, translate, and write code, which is powerful but also means its behavior is steered entirely by the text it reads. It has no built-in notion of which instructions are trusted, so any text in its input, including a web page or document it is asked to read, can try to influence it. It can also state false things confidently, called hallucination, so important output must be verified.

TA note: LLMs have no persistent memory between calls by default, no concept of who is trusted, and can hallucinate. Ask: if an LLM reads a web page and that page contains hidden instructions, can those instructions change the LLM output? Yes. That is indirect prompt injection.

Prompt injection

Hijacking an AI with text

Prompt injection is the top security risk for LLM applications. It works by hiding instructions inside content the model processes so that the model ignores its own rules and does what the attacker wants, such as revealing a secret or taking an unsafe action. Direct injection comes from the user message, while indirect injection hides instructions in external data the model reads, like a poisoned web page. Defenses include separating trusted instructions from untrusted data, filtering and validating inputs, limiting what the model is allowed to do, and adding output checks. No defense is perfect yet, which is why this remains an active and serious problem.

TA note: Have students try Lakera Gandalf right now. The goal is to get the AI to reveal its password. The AI has a system prompt telling it to keep the password secret. Students are injecting into the user turn. This makes prompt injection concrete immediately.
