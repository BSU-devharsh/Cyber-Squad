Encoding, encryption, and hashing

Three jobs that are easy to confuse

These three transform data but for completely different reasons. Encoding, such as Base64, reformats data into a safe character set so it survives transport; it uses no key and anyone can reverse it, so it provides no secrecy. Encryption scrambles data with a secret key so that only someone with the right key can read it; this is real confidentiality. Hashing produces a one-way fingerprint that cannot be reversed and is used to check integrity and to store passwords. Confusing them is a classic beginner mistake: Base64 looks scrambled but protects nothing, and a hash protects integrity but can never be turned back into the original.

TA note: The biggest misconception is that Base64 is encryption. Show students: base64.b64decode('SGVsbG8h').decode() returns Hello! Anyone can do that with zero key. Encoding is a format, not a secret.

Why does cryptography exist?

The problem of private communication over a public channel

Whenever two parties want to communicate privately over a channel that others can see -- a road, a radio frequency, the internet -- they face the same problem: an eavesdropper can intercept the message. Cryptography solves this by transforming the message so that only the intended recipient can read it. The word comes from Greek: kryptos (hidden) + graphein (to write). It has been used for at least 3,500 years, from Egyptian hieroglyphic substitutions to World War II cipher machines. Today it underpins every HTTPS connection, every encrypted file, and every digital signature.

TA note: Frame this historically. Caesar used it to send military orders. Enigma encrypted Nazi communications. Turing breaking Enigma may have shortened WW2 by two years. The math has changed but the need is identical.

Plaintext, ciphertext, and keys

The three terms in every crypto discussion

Three terms appear in every cryptography discussion. Plaintext is the readable original message. Ciphertext is the scrambled version produced by the encryption algorithm. A key is the secret input that controls how the scrambling works -- it is what separates a cipher from a simple code anyone can reverse. The same algorithm with a different key produces completely different ciphertext from the same plaintext. Security comes from the key staying secret, not from the algorithm being secret. This principle -- called Kerckhoffs's principle -- is why strong modern ciphers publish their algorithm openly.

TA note: Write the equation: encrypt(plaintext, key) to ciphertext. decrypt(ciphertext, key) to plaintext. Ask: what happens if the key leaks? The ciphertext is immediately readable. The key is everything.

The Caesar cipher

The first substitution cipher

Named after Julius Caesar, this cipher shifts every letter by a fixed amount, so with a shift of three the letter A becomes D. It is a substitution cipher: each letter is replaced by another according to a simple rule. Its fatal weakness is a tiny keyspace of only twenty-five useful shifts, so a computer simply tries them all in an instant, a method called brute force. Even by hand it falls to frequency analysis, because the most common ciphertext letter usually maps to the common English letter E. Modern ciphers fix this by using enormous keys that cannot be exhausted.

TA note: Have students try to break KHOOR by hand first (shift each letter back by 3). Then show how a computer tries all 25 shifts in milliseconds. Ask: how many shifts would a 256-bit key have? The answer is 2^256, which is more atoms than the observable universe.

Base64 and XOR

Encoding and a cipher building block

Base64 represents binary data using sixty-four printable characters so it can travel safely through email and URLs; the trailing equals signs are padding, and it is fully reversible by anyone. XOR, the exclusive-or operation, combines data with a key one bit at a time and has a neat property: applying the same key twice returns the original. That reversibility makes XOR a building block of real stream ciphers, which generate a long, unpredictable keystream and XOR it with the message. When the key is truly random, as long as the message, and never reused, XOR becomes the unbreakable one-time pad.

Hashing and the avalanche effect

Sensitivity and collisions

A strong hash shows the avalanche effect: changing a single bit of the input changes about half of the output bits unpredictably, so password and Password produce completely different fingerprints. Two further properties matter: preimage resistance means you cannot work backward from a hash to an input, and collision resistance means you cannot find two different inputs with the same hash. When a hash function breaks, those guarantees fail; MD5 and SHA-1 are now retired because collisions can be produced, while SHA-256 remains the safe default. These properties are what let hashes protect downloads, passwords, and digital signatures.

TA note: Run hashlib.sha256(b'password').hexdigest() and hashlib.sha256(b'Password').hexdigest() live. Count how many hex digits differ. Usually more than half change from one capital letter. This is what makes hashing tamper-evident.

Symmetric versus public-key encryption

Solving the key-sharing problem

Symmetric encryption, such as AES, uses one shared secret key to both encrypt and decrypt; it is fast, but both sides must somehow share the key securely first, which is hard over an open network. Public-key (asymmetric) encryption, such as RSA, solves this with a pair of keys: a public key that anyone may use to encrypt, and a private key that only the owner holds to decrypt. Its security rests on math that is easy one way and practically impossible to reverse, like multiplying two large primes versus factoring the result. In practice systems use both: public-key crypto to safely agree on a shared key, then fast symmetric crypto for the bulk data, which is exactly how HTTPS works.

HTTP versus HTTPS

Encrypting the web

HTTP, the HyperText Transfer Protocol, sends web requests and responses as plain text, so anyone watching the network can read or change them. HTTPS adds Transport Layer Security (TLS), which encrypts the connection and verifies the server identity with a certificate signed by a trusted authority. The TLS handshake uses public-key cryptography to agree on a fresh symmetric session key, then switches to fast symmetric encryption for the actual data. This stops eavesdropping and man-in-the-middle tampering, which is why the padlock and HTTPS are required for logins, payments, and any private data.

TA note: In Wireshark you will literally see the text of an HTTP login in the packet. For HTTPS it looks like random bytes. This is the most visceral demo of why HTTPS matters. Have students look for a username or password in the HTTP packet capture.

Frequency analysis

Breaking a cipher without the key

E is the most common English letter. Counting letters hints at the shift or substitution.

The Vigenere cipher

A stronger classic

A keyword varies the shift per letter, defeating simple frequency analysis. Still breakable with enough text.

AES

The workhorse

The Advanced Encryption Standard is fast, trusted symmetric encryption used in HTTPS and disk encryption.

RSA

Public-key math

RSA builds a key pair from two large primes. Encrypting is easy; factoring the result is practically impossible.

Digital signatures

Proving who sent it

Sign by hashing a message and encrypting the hash with your private key; anyone verifies with your public key.
